## 1. Step 별 변경 명세

### Step 0–1 — 초기 설정 & 경쟁사 매핑
- **상태**: 유지 (v4.0 구조 안정)
- **추가**: 횡단 원칙인 "출처 교차 검증" 안전장치 — 경쟁사 시총·티커 충돌 시 `[검증 필요]` 태그.

### Step 2 — 재무 분석
- **추가**: 🔤 **용어 번역기 모드**
  - 출력에 CAPEX·OPM·ROE·ROIC 등 등장 시 괄호 안 10자 내외 쉬운 풀이 자동 부착.
  - 예: `ROE(자기자본 대비 이익률)`
- **추가**: 수치 충돌 시 `[검증 필요]` 태그.

### Step 3 — 산업 분석
- **추가**: 🪄 **핵심 비유 한 줄** (일상 서비스에 빗댐)
  - 예: "이 회사는 반도체 업계의 '스타벅스'다 — 수많은 매장(Fab)을 직접 운영"
- **추가**: 📐 **Mermaid 밸류체인 다이어그램**
  - `flowchart LR` 코드 블록으로 출력 → 부원이 mermaid.live 또는 Notion 등에 붙여넣어 시각화.

### Step 4 — 기업 분석
- **추가**: 🔤 용어 번역기 모드 (Step 2와 동일).
- **추가**: 📐 Mermaid 지배구조 / 사업부 구조 다이어그램.

### Step 5 — 투자포인트 & 리스크
- **추가**: 🧪 **Sanity Check (자체 점검)**
  - "매출 ↑ → 주가 ↑" 같은 1차원 논리 자동 적발.
  - 적발 시 → 마진 개선 / 점유율 확대 / 멀티플 리레이팅 등 **중간 단계** 보강 후 재출력.

### ✨ Step 5.5 (신규) — Red Team
- **신설**: 가상의 **Short Seller 페르소나** 부여.
- 절차:
  1. Step 5에서 도출된 투자포인트의 논리적 허점 / 재무 데이터 맹점을 **3가지 공격 질문**으로 정리.
  2. 각 공격에 대한 **방어 논리** 작성.
  3. 방어 불가능한 항목은 Step 5 투자포인트에서 제외 또는 약화 표기.
- 효과: 리포트 신뢰도 + 논리적 깊이 향상.

### Step 6 — 밸류에이션
- **추가**: 🔤 용어 번역기 (PER·EV/EBITDA·DCF·WACC 등).
- **추가**: 동일 지표 간 수치 충돌 시 `[검증 필요]` 태그.

### Step 7 — 검토 & 수정
- **유지 + 강화**: 교차 검증 체크리스트에 Red Team 결과 반영 항목 추가.

### 🚨 Step 8 — 패러다임 전환 (전면 재작성)
- **금지**: PDF / PPTX / DOCX 직접 생성, 다운로드 링크 생성, 스크립트 실행.
- **새 출력 양식**: 슬라이드별 **텍스트 박스 단위 복붙 가이드**.
  ```
  [1페이지: 커버 슬라이드]
  - 메인 타이틀: [기업명] - [투자포인트 1줄 요약]
  - 투자의견 박스: BUY / 목표주가 ###,###원
  - 하단 표 데이터: (3개년 매출/영업이익 마크다운 표)

  [2페이지: 산업 분석]
  - 좌측 본문 (≤150자): [내용]
  - 우측 그래프 위치: '글로벌 시장 규모 추이' 캡처본 삽입
  ...
  ```
- **이점**:
  - 동아리 공식 PPT 템플릿 디자인 보존.
  - 글자 수 제한을 프롬프트 단계에서 강제 → 텍스트 박스 삐져나감 방지.
  - 비전공자도 "복붙 위치"만 따라가면 됨.

## 2. v5.0 횡단 원칙 (모든 Step 공통)

1. **100% 웹 챗봇 호환** — 외부 스크립트·코드 실행 의존 금지.
2. **두 페르소나 동시 커버**
   - 금융 知 ↑ / 프롬프트 ↓ — 용어가 익숙한 부원이 빠르게 검수할 수 있도록 구조 일관성 유지.
   - 프롬프트 知 ↑ / 금융 ↓ — 용어 번역기 + 비유 + Sanity Check로 이해 보조.
3. **3중 신뢰도 검증** — Sanity Check (자체) → Red Team (반박) → Step 7 (재검토).
4. **시각화 내장** — Mermaid 코드 블록으로 다이어그램 즉시 출력 (외부 디자인 도구 불필요).
5. **출처 교차 검증** — 의심 수치는 `[검증 필요]` 태그 표시.

## 4. 작업 순서 제안

1. Step 8 프롬프트 초안 작성 (가장 임팩트 큼, 실패 케이스가 가장 명확).
